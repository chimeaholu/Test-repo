# RB-070 Load Test Report

## Scope

RB-070 covered realistic API load against the current AgroDomain application surfaces with the scenarios defined in the R8 PRD:

- 100 concurrent users browsing marketplace listings
- 50 concurrent users creating listings
- 20 concurrent users executing negotiation workflows
- 10 concurrent users performing wallet operations
- Mixed 200-concurrent workload across marketplace, farm, climate, wallet, fund, transport, and health endpoints

## Methodology

- Seeded an isolated benchmark database per run with `tests/load/prepare_benchmark_db.py`.
- Executed the API locally through `tests/load/run_rb070.py`, which:
  - migrated and seeded a fresh SQLite benchmark database,
  - started the API on `127.0.0.1:8010`,
  - ran the marketplace and mixed scenarios via `tests/load/harness.mjs`,
  - sampled process-tree RSS before, during, and after the scenarios.
- Harness settings:
  - request timeout: `5000ms`
  - warmup window: `3s`
  - runtime for marketplace phases: `18s` each
  - runtime for mixed phase: `24s`
- Functional regression validation after the performance fixes:
  - `pytest apps/api/tests/integration/test_wallet_transfer_routes.py apps/api/tests/integration/test_wallet_escrow_runtime.py apps/api/tests/unit/test_wallet_ledger_and_escrow.py -q`
- Targeted contention repro:
  - 200 concurrent `POST /api/v1/wallet/transfers` requests against the same sender/recipient pair before the wallet fix
  - same repro repeated after the wallet fix

Observed failure mode:

- the final pooled Postgres run did not fail on migration or seeding after the in-lane fixes
- single-request smoke traffic succeeded

Residual blocker:

- the current production-like storage profile available in this runtime, specifically the Supabase transaction pooler path at `aws-1-us-east-1.pooler.supabase.com:6543`, does not sustain the RB-070 concurrency envelope with the current sync API/runtime shape
- this means RB-070 remains blocked not by SQLite anymore, but by pooled Postgres throughput collapse under benchmark load

Conclusion:

- full benchmark execution on a stronger concurrent-write backend was completed
- it does **not** clear the final R8 blocker
- the next exact step is to rerun the same harness against direct Postgres connectivity from an IPv6-capable or colocated runtime, then optimize remaining hot paths only if that direct profile still misses target

## Findings

### 1. Initial deployment shape was the first bottleneck

The baseline run timed out almost completely because the API was effectively running as a single uvicorn worker. Under the RB-070 workload, that configuration could not absorb concurrent reads and writes.

Fix applied:

- added `apps/api/start.sh`
- updated `Procfile`
- updated `apps/api/Dockerfile`

The launcher now runs multiple uvicorn workers by default, with `WEB_CONCURRENCY` override support.

### 2. Several read paths were still doing avoidable fan-out work

The mixed workload exposed repeated per-record follow-up reads when assembling farm, climate, wallet, fund, transport, and marketplace payloads.

Fixes applied:

- batched negotiation message reads
- batched farm fields, activities, inputs, and crop cycles
- batched climate farm-profile and observation lookups
- batched escrow timeline lookups
- batched fund opportunity lookups
- batched transport shipment lookups
- batched marketplace published-revision lookups for browse traffic
- SQLite temp-store/cache tuning and targeted read-order indexes across marketplace, negotiation, farm, and climate hot paths
- WAL/checkpoint tuning for the write path

Impact:

- browse phase improved from `80.13%` errors / `p50 5012.7ms` / `p99 5280.2ms` in `fixed3` to a best observed `0.00%` errors / `p50 986.0ms` / `p99 2359.6ms` in `fixed5`
- create-listings improved from `8.88%` errors / `p99 5060.2ms` in `fixed3` to a best observed `0.39%` errors / `p99 3922.9ms` in `fixed6`
- mixed workload error rate improved from `27.84%` in `fixed3` to a best observed `14.77%` in `fixed5`
- wallet improved from `0.10%` errors / `p99 819.4ms` to a best observed `0.00%` errors / `p99 339.9ms`

### 3. Wallet transfer 500s were a real concurrency bug

The final harness-clean run still showed wallet 500s. A targeted repro isolated the cause:

- before fix: 200 concurrent transfers against one sender/recipient pair returned `45x 200` and `155x 500`
- server logs showed `sqlite3.IntegrityError: UNIQUE constraint failed: wallet_ledger_entries.wallet_id, wallet_ledger_entries.entry_sequence`

Root cause:

- concurrent requests could compute the same `entry_sequence` for the same wallet before either transaction committed

Fix applied:

- added sender/recipient wallet file locks in `apps/api/app/api/routes/wallet.py`
- kept bounded retry handling for transient SQLite write conflicts

Validation:

- after fix: the same 200-concurrent repro returned `200x 200`
- final load run removed all 5xx responses across every RB-070 phase

### 4. The remaining blocker is long-tail timeout pressure in the current SQLite runtime

The biggest remaining failures are now create-listings tail latency, negotiation p99, and mixed 200-concurrent tail latency/error budget. The dominant final failure mode is still request timeout, not validation or 5xx behavior.

Evidence:

- browse marketplace is materially better, but `p99` is still `2359.6ms`
- create-listings `p99` is still `5002.2ms`
- negotiation `p99` is still `3785.0ms`
- mixed workload `p95` is `5012.1ms` and `p99` is `5383.4ms`
- failure buckets are dominated by aborted requests at the `5s` timeout ceiling

Inference:

- the current single-node, SQLite-backed deployment is still not sufficient for the PRD tail-latency target once high-concurrency mixed traffic and command writes are combined
- the median path is much healthier after the final read-path fixes, but the current runtime still cannot keep the `p99 < 2000ms` and zero-error guarantees under the RB-070 workload

## Gate Readiness Notes

RB-070 is improved materially, but the lane is still not ready to claim full PRD pass.

What is ready:

- realistic load harness and seeded benchmark data exist
- final run is free of 5xx responses
- wallet operations are stable under concurrency after the transfer-lock fix and now meet the target
- browse-marketplace is no longer the primary blocker after the published-revision batching fix
- additional SQLite-native tuning reduced browse tail and roughly halved mixed-workload error rate
- database connections remained operational for the duration of the run
- successive safe SQLite tunings improved different phases, but no tuning combination got all phases under the RB-070 bar

What is not ready:

- browse-marketplace still misses the `p99 < 2000ms` target
- create-listings does not meet p99 or zero-error targets
- negotiation does not meet the p99 target
- mixed 200-concurrent traffic does not meet latency or zero-error targets

Go/No-Go recommendation for R8 closure:

- `NO-GO` for marking RB-070 green on the current single-node SQLite runtime.
- To close R8 cleanly, either:
  - approve an explicit performance waiver for SQLite-only benchmark conditions, or
  - move the benchmark/runtime storage profile to a production-like architecture with stronger concurrent write behavior, such as Postgres or equivalent
